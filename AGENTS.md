## Learned User Preferences

- Ask the user for permission to run the application.
- Use `./run_tests.sh` as the canonical way to run the test suite when validating changes.
- Run scripts individually (e.g. `./docker-compose.sh up`, `startup.sh`); avoid suggesting ad-hoc chained commands that paste multiple script names into one shell line unless the user requests that.
- Standard app port for this repo is 5001 everywhere (Docker and local); port 5000 is intentionally avoided because it is always busy on the user's machine.
- Commit messages should be short and concise, sentence-style, with commas separating the main changes (not a long paragraph or bullet list in the subject).
- When the user states they can restart fresh, prefer straightforward changes over extra compatibility layers unless they explicitly want to preserve old behavior.
- Do not present `sh -n` on shell files as a project workflow; it only parses syntax and does not run or start anything.
- Prefer minimal, essential code comments; avoid narrating what the code already states; when simplifying staged changes, remove unused imports.
- Keep `.env.example` concise: short section headers and one-line hints rather than long explanatory comments.
- After backend dependency updates, run `pip install -r requirements.txt` in the project venv before tests, and keep `requirements.txt` focused on dependency pins (no long commentary blocks unless requested).
- Serialize API fields that represent instants in time as RFC 3339 / ISO 8601 with an explicit `Z` or numeric offset so clients (e.g. JavaScript) do not parse naive strings as local time; do not add OpenAPI/Swagger `license` metadata unless a matching root `LICENSE` file exists (`license` in docs is not a substitute for repo licensing).
- For operational guidance, give brief context first, then a short numbered to-do list with concrete UI paths and real known URLs; avoid vague placeholders, avoid overly obvious setup steps, and prefer direct recommendations over ambiguous option lists.

## Learned Workspace Facts

- Bring up with `./docker-compose.sh up` or `startup.sh`. `.env` is in `.dockerignore`; Compose still reads the host `.env` for substitution and passes variables listed under the service `environment:` block. REST Client `requests.http` may need `Accept: application/json` where curl succeeds but the client gets 403.
- `GET /api/ktc/rankings` is DB-backed; KTC scrapes use refresh endpoints. With `REDIS_URL`, that GET uses shared Redis (`cache/redis_rankings.py`, TTL in `cache/settings.py`); on Vercel Production (`VERCEL_ENV=production`), `REDIS_URL` is required for that shared cache path. Logs include a safe Redis URL and hit/miss; warm keys after KTC data exists via `./docker-compose.sh warm-cache`. In containers, use the Redis service hostname or `rediss://`, not `localhost` unless Redis is colocated. Response includes `X-Rankings-Cache` (`HIT` from in-process or Redis, `MISS` when loaded from Postgres and cached).
- Gunicorn binds `0.0.0.0:5001` only; an extra `[::]:5001` bind previously broke host reachability. `app.py` enables Flask-Compress for gzip on JSON.
- `GET /api/dashboard/league/<id>` includes `researchMeta` (camelCase). Prefer DB `last_updated` for freshness; `timestamp` is when the JSON was built. UTC ISO on server. Postgres `timestamp without time zone` with a non-UTC DB session timezone can shift how those values read (e.g. an hour ahead); SQLAlchemy engines use `connect_args` with `options` `-c timezone=UTC` in `app.py` and `vercel_app.py`. With `REDIS_URL`, checks Redis before loading the full league snapshot; cache key `dashboard:league:v1:{league_id}:{season}:{league_format}:{tep}:{is_redraft}`; headers `X-Dashboard-League-Cache` (`HIT`/`MISS`), `X-Dashboard-League-Payload-Bytes`. Miss path logs segment timings including `ms_run_players`, `ms_json`, `ms_redis_*`, research subqueries; each `ms_*` field is elapsed time in milliseconds. `DASHBOARD_LEAGUE_REDIS_TTL_SECONDS` default 86400 (24h); invalidate via league refresh / daily refresh and KTC rankings invalidation. Player rows are loaded via `selectinload` for the requested format only (no KTC N+1) and serialized through a slim dashboard DTO that skips full `Player.to_dict()`.
- On Vercel, `vercel_app.py` resolves the DB URL from the first set variable among `POSTGRES_URL`, `POSTGRES_PRISMA_URL`, `DATABASE_URL`, and `POSTGRES_URL_NON_POOLING`. Local `app.py` uses `DATABASE_URL` via `utils/constants` (`TEST_DATABASE_URI` overrides for tests); it does not read `POSTGRES_URL` first. Supabase dashboard “API requests” often reflect REST/Auth HTTP traffic, not every direct SQLAlchemy/psycopg2 session to Postgres—use Database-oriented metrics when checking whether the app is hitting that project.
- Cron-gated `/api/maintenance/nightly-sync` (and `/prewarm`, same prewarm step) require `Authorization: Bearer <CRON_SECRET>` when `VERCEL_ENV=production` (header-only auth is disabled in production). Manual runs use the same `GET` or `POST` route and Bearer. Neither runs slow Sleeper NFL ingest. Pipeline: KTC formats → leagues → research; research seasons come from each league API `season`; if `skip_leagues` and no `seasons`, research skips. `POST /api/sleeper/refresh` (60s+) stays operator-only and out of scheduled runs. `GET /api/maintenance/health` documents paths and auth only—it does not run the pipeline. For production, call maintenance on the deployed backend origin (same host as `/api/...`), not the Supabase REST hostname. **Vercel:** `vercel.json` has one daily cron `30 15 * * *` UTC; Hobby allows at most one run per day per job.
- `GET /api/dashboard/league/<id>` may attach per-player `stats` (`average_points`, `total_points`, `games_played`) from `SleeperWeeklyData` rows, aggregated over weeks 1–17 (week 18 excluded; see `SLEEPER_STATS_AGGREGATE_WEEK_*` in `utils/constants.py`). The handler never scrapes weekly stats inline; missing rows are populated by the nightly cron (`refresh_weekly_stats_for_leagues`) or by `scripts/seed_three_leagues.py`. Because `SleeperWeeklyData` is keyed by `(season, week, league_type, player_id)` (not `league_id`), once any one league has been refreshed for a season/league_type, every other league reads the same per-player stats. **`nightly-sync`** optionally runs **`_prewarm_dashboard_caches`** after the pipeline; **`skip_prewarm`** opts out.
- Schema cleanup for duplicate height/birth fields: `sql/migrations/20260426_drop_duplicate_player_fields.sql` drops legacy KTC columns `heightFeet`, `heightInches`, and `birthday` on `players` in favor of Sleeper `height` and `birth_date`. Postgres `UndefinedColumn` on those legacy names in production usually means migrations and deployed code are out of step (old build still selecting dropped columns or DB not migrated yet). Sleeper `position` `RDP` (rookie/draft pick rows per `SLEEPER_POSITION_RDP` in `utils/constants.py`) is valid data to persist; `utils/player_eligibility.py` and scrapers treat it like other roster positions—do not strip or treat it as invalid when saving.
- After Sleeper saves, stale `Player` rows not in the export are pruned (ORM cascade). Indexed `Player.sleeper_player_id` and weekly-research indexes matter; existing Postgres may need `sql/ddl_performance_indexes.sql`. `sleeper_rosters` stores roster JSON in `roster_metadata` (not SQLAlchemy `metadata`); `sleeper_users` uses `user_metadata` as source of truth for `team_name` with legacy column fallback. Optional `KTC_EXPORT_JSON_AND_S3` (default off): when truthy, KTC refresh may write JSON and upload to S3 if `S3_BUCKET` is set. Persisted players should have a `match_key` and exclude Sleeper placeholder `search_rank` `9999999`; use `scripts/cleanup_invalid_players.py` (dry run by default; `--execute` does batched ORM deletes so child KTC rows cascade) or SQL under `sql/migrations/` for legacy bad rows. With both `TEST_DATABASE_URI` and `DATABASE_URL` in `.env`, the script loads `.env` then unsets `TEST_DATABASE_URI` so the target DB matches production, not pytest SQLite. KTC merge paths preload eligible Sleeper rows from the DB (`load_sleeper_players_for_merge_from_db` in `scrapers/pipelines.py`) to avoid redundant full-table reads when refresh passes them through.
- Backend Docker image uses Debian slim (glibc), e.g. `python` slim bookworm, so PyPI manylinux wheels (e.g. `rpds-py`) install reliably; Alpine/musl is avoided for that reason. Test databases may be unseeded; backend `pytest` can fail if wheels target the wrong CPU architecture; reinstall dependencies in a native-architecture venv after `pip install -r requirements.txt`.
- Flask: register blueprints via `routes/registry.py`; `routes/__init__.py` stays empty. Packages use empty `__init__.py` files and concrete imports from `models/`, `managers/`, `scrapers/`, `routes/ktc`, `routes/sleeper`, `cache/`, `services/`, `utils/`; dashboard route module is `routes/dashboard_league.py`; DDL reference `sql/ddl_performance_indexes.sql`; entrypoints include `app.py`, `vercel_app.py`, `wsgi.py`, `gunicorn.conf.py`. Many routes return validation and client errors via `json_api_error()` in `routes/helpers.py` (`status`, `error`, RFC 3339 `timestamp`, optional `details`); unexpected exceptions on `@with_error_handling` handlers are 500 with the same envelope. Keep `openapi.yaml`, `routes/swagger_config.py` examples, and `README.md` aligned with live prefixes (e.g. league refresh is `POST /api/sleeper/league/{id}`, not `.../refresh`). `POST`/`PUT` `/api/sleeper/league/{id}` uses HTTP 404 when the Sleeper scrape fails and 500 when persistence fails—not HTTP 200 with only `status` in the body.
- `sleeper-dashboard` polls `GET /api/ktc/refresh/status/{job_id}` until a terminal state before reloading the dashboard bundle; the refresh-then-immediately-load race is fixed.
- `sleeper_weekly_data` rows are keyed by `(season, week, league_type, player_id)` and intentionally serve two consumers: research (`owned`/`started` blob in `research_data`) and matchup points (`points`, `is_starter`, `roster_id`). Persist updates with **upsert** semantics—never delete the week before insert—so research refreshes do not clobber matchup points and vice versa. Both `routes/sleeper/research.py::_upsert_research_rows` and `services/daily_refresh.py::persist_research` follow this contract.
- Research retention follows the dynasty-window rule from the player-data-coverage plan: for `season == max(SleeperLeague.season)` the API honors `week=1..18` / `week=all`; for older seasons only **week 18** is refreshed (any other request is logged and truncated to `[18]`). The `refresh_results.retention_applied` field on `PUT/POST /api/sleeper/players/research/{season}` reports `requested` vs `prior-season-week-18`.
- The research payload union of keys is documented in `sleeper-backend/docs/sleeper-research-payload-keys.md`. Re-run `scripts/probe_sleeper_research_keys.py` before adding UI columns sourced from research data; today only `owned` and `started` are observed in production payloads, so `SleeperWeeklyData` columns like `projected_points`, `snap_count`, and weather fields remain placeholders.
- Trade analyzer `POST /api/trade-analyzer/preview` and `/analyze` build a slim LLM context (`services/trade_analyzer/context.py`): `market_*` on trade and roster players plus `league.research_week`; roster rows omit position/positional_tier (grouped by position), `name` only for trade players; picks use `label` + `ktc_value` only; `trade.side_*_outgoing` + `ktc_totals` only (no `trade_summary`, no duplicate incoming lists); nested trade `ktc` omits value/rank/trend duplicates; `starter_slots_required` on `league` only. Sample analyze uses `provider: echo` or `tests/fixtures/data/trade_analyzer_echo.json`. Payload reference: `docs/trade-analyzer-payload.md`.
