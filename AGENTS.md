## Learned User Preferences

- Ask the user for permission to run the application.
- Use `./run_tests.sh` as the canonical way to run the test suite when validating changes.
- Run scripts individually (e.g. `./docker-compose.sh up`, `startup.sh`); avoid suggesting ad-hoc chained commands that paste multiple script names into one shell line unless the user requests that.
- Standard app port for this repo is 5001 everywhere (Docker and local); port 5000 is intentionally avoided because it is always busy on the user's machine.
- Commit messages should be short and concise, sentence-style, with commas separating the main changes (not a long paragraph or bullet list in the subject).
- When the user states they can restart fresh, prefer straightforward changes over extra compatibility layers unless they explicitly want to preserve old behavior.
- Do not present `sh -n` on shell files as a project workflow; it only parses syntax and does not run or start anything.
- Prefer minimal comments and concise `.env.example` (short section headers, one-line hints); do not narrate what code already states or mention implementation phase numbers.
- After backend dependency updates, run `pip install -r requirements.txt` in the project venv before tests, and keep `requirements.txt` focused on dependency pins (no long commentary blocks unless requested).
- Serialize API fields that represent instants in time as RFC 3339 / ISO 8601 with an explicit `Z` or numeric offset so clients (e.g. JavaScript) do not parse naive strings as local time; do not add OpenAPI/Swagger `license` metadata unless a matching root `LICENSE` file exists (`license` in docs is not a substitute for repo licensing).
- For operational guidance, give brief context first, then a short numbered to-do list with concrete UI paths and real known URLs; avoid vague placeholders, avoid overly obvious setup steps, and prefer direct recommendations over ambiguous option lists.
- Use `python3` (not `python`) in terminal and `docker compose exec` command examples.

## Learned Workspace Facts

- Bring up with `./docker-compose.sh up` or `startup.sh`. Backend Docker image uses Debian slim (glibc) for reliable PyPI wheels; Alpine/musl is avoided. Test DBs may be unseeded; reinstall deps in a native-architecture venv after `pip install -r requirements.txt` if wheels target the wrong CPU. `.env` is in `.dockerignore`; Compose still reads the host `.env` for substitution and passes variables listed under the service `environment:` block. REST Client `requests.http` may need `Accept: application/json` where curl succeeds but the client gets 403.
- `GET /api/ktc/rankings` is DB-backed; KTC scrapes use refresh endpoints. With `REDIS_URL`, that GET uses shared Redis (`cache/redis_rankings.py`, TTL in `cache/settings.py`); on Vercel Production (`VERCEL_ENV=production`), `REDIS_URL` is required for that shared cache path. Logs include a safe Redis URL and hit/miss; warm keys after KTC data exists via `./docker-compose.sh warm-cache`. In containers, use the Redis service hostname or `rediss://`, not `localhost` unless Redis is colocated. Response includes `X-Rankings-Cache` (`HIT` from in-process or Redis, `MISS` when loaded from Postgres and cached).
- Gunicorn binds `0.0.0.0:5001` only; an extra `[::]:5001` bind previously broke host reachability. `app.py` enables Flask-Compress for gzip on JSON.
- `GET /api/dashboard/league/<id>` includes `researchMeta` (camelCase). Prefer DB `last_updated` for freshness; `timestamp` is when the JSON was built. UTC ISO on server. Postgres `timestamp without time zone` with a non-UTC DB session timezone can shift how those values read (e.g. an hour ahead); SQLAlchemy engines use `connect_args` with `options` `-c timezone=UTC` in `app.py` and `vercel_app.py`. With `REDIS_URL`, checks Redis before loading the full league snapshot; cache key `dashboard:league:v1:{league_id}:{season}:{league_format}:{tep}:{is_redraft}`; headers `X-Dashboard-League-Cache` (`HIT`/`MISS`), `X-Dashboard-League-Payload-Bytes`. Miss path logs segment timings including `ms_run_players`, `ms_json`, `ms_redis_*`, research subqueries; each `ms_*` field is elapsed time in milliseconds. `DASHBOARD_LEAGUE_REDIS_TTL_SECONDS` default 86400 (24h); invalidate via league refresh / daily refresh and KTC rankings invalidation. Player rows are loaded via `selectinload` for the requested format only (no KTC N+1) and serialized through a slim dashboard DTO that skips full `Player.to_dict()`. On cache miss for a new league, scrape league/rosters only and load merged player data from Postgres—do not inline-scrape the full Sleeper NFL players export.
- Local dev uses **Supabase** via `DATABASE_URL` in `.env` (not the docker-compose `postgres` service unless `DATABASE_URL` points there). On Vercel, `vercel_app.py` resolves the DB URL from the first set variable among `POSTGRES_URL`, `POSTGRES_PRISMA_URL`, `DATABASE_URL`, and `POSTGRES_URL_NON_POOLING`. Local `app.py` uses `DATABASE_URL` via `utils/constants` (`TEST_DATABASE_URI` overrides for tests). Apply SQL migrations and RAG ingest against the same DB `DATABASE_URL` uses—`docker compose exec postgres psql` targets the wrong DB when the app uses Supabase. Supabase dashboard “API requests” often reflect REST/Auth HTTP traffic, not every direct SQLAlchemy session—use Database-oriented metrics when checking whether the app is hitting that project.
- Cron-gated `/api/maintenance/nightly-sync` requires `Authorization: Bearer <CRON_SECRET>` when `VERCEL_ENV=production` (header-only auth is disabled in production). Manual runs use the same `GET` or `POST` route and Bearer. Neither runs slow Sleeper NFL ingest. Pipeline: KTC formats → leagues → research; research seasons come from each league API `season`; if `skip_leagues` and no `seasons`, research skips. `POST /api/sleeper/refresh` (60s+) stays operator-only and out of scheduled runs. `GET /api/maintenance/health` documents paths and auth only—it does not run the pipeline. For production, call maintenance on the deployed backend origin (same host as `/api/...`), not the Supabase REST hostname. **Vercel:** `vercel.json` has one daily cron `30 15 * * *` UTC (~11:30 AM ET); Hobby allows at most one run per day per job.
- `GET /api/dashboard/league/<id>` may attach per-player `stats` (`average_points`, `total_points`, `games_played`) from `SleeperWeeklyData` rows, aggregated over weeks 1–17 (week 18 excluded; see `SLEEPER_STATS_AGGREGATE_WEEK_*` in `utils/constants.py`). The handler never scrapes weekly stats inline; missing rows are populated by the nightly cron (`refresh_weekly_stats_for_leagues`) or by `scripts/seed_three_leagues.py`. Because `SleeperWeeklyData` is keyed by `(season, week, league_type, player_id)` (not `league_id`), once any one league has been refreshed for a season/league_type, every other league reads the same per-player stats. **`nightly-sync`** optionally runs **`_prewarm_dashboard_caches`** after the pipeline; **`skip_prewarm`** opts out.
- KTC refresh no longer clears Sleeper-owned player fields (e.g. `search_rank`, injury); dynasty and redraft KTC rankings are stored and served separately—rankings, dashboard bundle, and trade analyzer all respect `is_redraft`.
- Schema and player hygiene: `sql/migrations/20260426_drop_duplicate_player_fields.sql` drops legacy KTC `heightFeet`/`heightInches`/`birthday` in favor of Sleeper `height`/`birth_date`; Sleeper `position` `RDP` is valid (`SLEEPER_POSITION_RDP`). After Sleeper saves, stale `Player` rows not in the export are pruned (ORM cascade). Persisted players need a `match_key` and exclude placeholder `search_rank` `9999999`; use `scripts/cleanup_invalid_players.py` (dry run default) or SQL under `sql/migrations/`.
- Flask: register blueprints via `routes/registry.py`; `routes/__init__.py` stays empty. Packages use empty `__init__.py` files and concrete imports from `models/`, `managers/`, `scrapers/`, `routes/ktc`, `routes/sleeper`, `cache/`, `services/`, `utils/`; dashboard route module is `routes/dashboard_league.py`. Many routes return validation errors via `json_api_error()` in `routes/helpers.py` (`status`, `error`, RFC 3339 `timestamp`, optional `details`). Keep `openapi.yaml`, `routes/swagger_config.py` examples, and `README.md` aligned with live prefixes (e.g. league refresh is `POST /api/sleeper/league/{id}`).
- `sleeper_weekly_data` rows are keyed by `(season, week, league_type, player_id)` and serve research (`owned`/`started` in `research_data`) and matchup points (`points`, `is_starter`, `roster_id`). Persist with **upsert** semantics—never delete the week before insert. Research retention: for `season == max(SleeperLeague.season)` honor `week=1..18` / `week=all`; for older seasons only **week 18** is refreshed. Union of research keys is in `docs/sleeper-research-payload-keys.md`; today only `owned` and `started` are observed in production payloads.
- Trade analyzer `POST /api/trade-analyzer/preview` and `/analyze` build slim LLM context (`services/trade_analyzer/context.py`). Production routing locks to **Gemini** (`policy.py`); dev defaults to Ollama unless overridden. Response schema (`output_schema.py`): `winner`, two `summary_bullets`, and `side_a`/`side_b` each with `trade_grade` (A+ through F−), `pros`, `cons`, and `value_delta` (not legacy `fairness_score` or `ktc_delta`). Sample analyze uses `provider: echo` or `tests/fixtures/data/trade_analyzer_echo.json`. Payload reference: `docs/trade-analyzer-payload.md`. **RAG (Phase 4 v1):** behind `TRADE_ANALYZER_RAG` (default off); strategy KB + feedback corpora in Supabase pgvector (`rag_documents`, `sql/migrations/20260609_add_rag_pgvector.sql`); ingest via `python3 scripts/rag_ingest_strategy.py` and `rag_ingest_feedback.py`. Local RAG needs `TRADE_ANALYZER_RAG=1`, Ollama `nomic-embed-text`, and `OLLAMA_HOST=http://host.docker.internal:11434` in Docker (not `localhost`). Vercel with `TRADE_ANALYZER_RAG=1` **fails open**—analyze works but retrieval is skipped until a cloud embedding provider exists (only Ollama implemented today).
